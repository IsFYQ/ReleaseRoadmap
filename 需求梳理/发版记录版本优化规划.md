## 二、现状评估

### 2.3 数据流现状


### 2.4 V4 规划对照

- **完全未涉及**：商业地图、后台管理流程、运营维护系统化设计

## 三、方向 A：后台管理模块

### 3.1 关键决策（2026-07-06 已确认）

以下功能方向已与用户确认，无需再沟通：

| 决策项 | 结论 | 影响 |
|--------|------|------|
| 钉钉文档拉取 | ✅ 可以自动从钉钉文档拉取 | 后台需集成钉钉文档 API，自动同步功能点数据 |
| 发版详情产出流程 | 先生成**预览网页** → 王蓉确认 → 一键**导出图片** | 后台需含网页预览 + HTML-to-图片导出功能 |
| 多业务线支持 | 直播教学运营**后续要接入** | 权限模型需按业务线隔离，预留扩展能力 |

### 3.1.1 运营现状确认（2026-07-06）

与王蓉确认后的实际运营流程：

| 项目 | 现状 | 对后台设计的影响 |
|------|------|------------------|
| 钉钉文档 | 公司内公开，表格格式 | 后台需解析钉钉表格，自动提取系统/功能入口/功能点 |
| 长图生成 | 通过 AI 自动生成 | 后台预览网页样式需与现有长图保持一致 |
| 截图来源 | 王蓉自己截取，自行添加红色箭头标注 | 后台需支持上传截图，保留现有标注样式 |
| 原始素材 | 存放在钉钉文档中 | 截图存储方案需考虑与钉钉文档的集成 |
| 文字稿 | **不存在**，直接生成图片 | 后台需提供结构化录入界面，替代文字稿环节 |
| 最终产出 | 图片发送到钉钉群 | 后台需支持一键导出图片并方便发送到钉钉群 |

**关键用户场景**：
```
王蓉打开后台 → 输入钉钉文档URL → 系统自动拉取并解析表格 → 
自动填充系统/功能入口/功能 → 王蓉补充是否重要功能、上传截图 → 
点击「预览」→ 新窗口打开预览网页（与最终图片样式一致）→ 
确认无误 → 点击「下载」→ 下载PNG长图 → 发送到钉钉群
```

**后台核心功能需求**：
1. **自动填充**：系统（产品线）、功能入口、功能点（从钉钉文档拉取）
2. **手动补充**：是否重要功能、截图
3. **预览**：生成预览网页，与最终图片样式一致
4. **导出**：一键下载图片（PNG长图）

### 3.2 技术方向：独立 Web 后台

**核心定位**：在看板项目内做一个独立的发版录入界面，运营直接使用，不再依赖开发拉钉钉。

#### 3.2.1 数据结构扩展
在 `release_data.js` 的 item 结构上新增字段：

```javascript
{
  date, pl, fp, fe, type,           // 现有
  module, scope, special, limit,    // 从 parsed_data_full.json 回流
  is_highlight,                      // 是否亮点
  verify_conclusion,                 // 运营验证结论（新）
  images: [],                        // 图片URL数组（新）
  release_id,                        // 单次发版唯一ID（新）
  environment,                       // 端（PC/App/小程序，从scope细化）
  operator,                          // 录入人（新）
  created_at, updated_at             // 审计字段（新）
}
```

#### 3.2.2 后台功能模块

| 模块 | 功能 | 优先级 |
|------|------|--------|
| **钉钉文档同步** | 自动拉取钉钉文档功能点数据，解析并导入后台 | P0 |
| **发版录入表单** | 日期/产品线/类型/功能点/入口/图片上传/特殊说明 | P0 |
| **发版列表管理** | 按日期分组/搜索/筛选/编辑/删除 | P0 |
| **预览网页生成** | 实时生成发版详情预览页，王蓉确认后再导出 | P0 |
| **图片导出** | 一键将预览页导出为 PNG/JPG 长图 | P0 |
| **图片上传与存储** | 本地存储或OSS，生成可访问URL | P0 |
| **权限管理** | 按业务线隔离数据，闻道作业/直播教学运营分权 | P0 |
| **多用户管理** | 支持多运营协同录入，按业务线分配权限 | P1 |
| **数据导出** | 导出为JSON/Excel供看板消费 | P1 |

**关键流程**：
```
钉钉文档 → 后台自动同步 → 运营调整补充 → 生成预览网页 → 确认无误 → 一键导出图片
```

#### 3.2.3 技术选型方向（待沟通后最终确认）

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **本地 Flask+SQLite** | 轻量、与现有Python脚本一致、可双击启动 | 仅本地访问、不支持多人协同 | 王蓉单人维护 |
| **Flask+JSON文件** | 极轻量、数据透明可手工编辑 | 并发写入风险 | 过渡方案 |
| **完整Web应用（Flask/Node+DB）** | 支持多人、可部署、可接入刘菡系统 | 开发量大、需运维 | 接入刘菡系统后 |

**建议**：先以"本地 Flask+SQLite"启动（1-2周可上线），等接入刘菡系统时再评估是否升级为完整 Web 应用。

**关键技术组件**：
1. **钉钉文档 API 集成**：用于自动拉取功能点数据，需申请钉钉开放平台权限
2. **HTML-to-Image 导出**：使用 Puppeteer/Playwright 或类似工具，将预览网页转为图片
3. **权限系统**：基于角色的访问控制（RBAC），按业务线（闻道作业/直播教学）隔离数据

#### 3.2.4 与看板的集成
- 看板继续以 HTML 双击打开为优先形态
- 后台导出 `release_data.js`，看板读取（保持现有内联 JS 模式）
- 单次发版详情页可作为独立 HTML，从看板条目点击跳转

## 四、方向 B：闻道作业商业地图

### 4.0 现状前提
- **商业地图现状**：Excel/结构化清单 → 第一阶段需转为 HTML 可视化
- **环节定义来源**：从现有材料抽取 → 需先定位"环节清单"在哪份文档
- **候选材料**：`参考资料/项目映射表.xlsx`、季度规划文档、TAPD 中"闻道作业商业地图"需求（owner: 张颖）

### 4.1 第一阶段：商业地图线上化

**目标**：将 Excel 结构化清单转为 HTML 可视化地图，嵌入看板作为新 Tab（M8 商业地图）。

**待办**：
1. 定位环节清单源文件（候选：项目映射表.xlsx / 季度规划 / TAPD 需求）
2. 抽取环节层级结构（如：获客→教学→作业→评测→数据→管理）
3. 设计地图可视化形式：
   - 选项 A：分层树状图（环节→子环节→功能模块）
   - 选项 B：流程图（按业务流程横向铺开）
   - 选项 C：矩阵图（环节×产品线）
4. 实现 HTML 渲染，支持点击节点查看详情
5. 作为新 Tab 接入主看板

**产出**：`商业地图.html`（或集成进 `产品看板.html` 的 M8 模块）

### 4.2 第二阶段：发版内容×环节结合

**目标**：选择某个环节，能看到该环节共发了几次版、最近做了什么更新。

**关键工作**：

1. **环节匹配**：对现有发版需求做环节标注
   - 输入：`release_data.js` 全量发版记录
   - 输出：每条记录新增 `environment`（环节）字段
   - 方法：先人工标注种子集 → 提炼关键词规则 → 自动匹配 + 人工校验

2. **环节维度视图**：
   - 环节卡片：每个环节显示发版次数 / 最近更新时间 / 主要功能点
   - 环节详情页：点击环节→展开该环节的所有发版记录（时间倒序）

3. **数据回流**：匹配结果存入 `release_data.js` 的 `environment` 字段

**待沟通**：环节清单由谁最终拍板？（建议张颖，作为商业地图需求 owner）

### 4.3 第三阶段：环节完成度评估

**目标**：选择环节即可评估整体完成度，方便对学校制作解决方案。

**关键工作**：

1. **完成度算法设计**：
   - 每个环节定义"预期功能点清单"（来自商业地图规划）
   - 实际已发版功能点 / 预期功能点 = 完成度
   - 可加权（核心功能权重高）

2. **解决方案生成器**：
   - 输入：学校类型（小学/初中/高中）+ 选中环节
   - 输出：该环节完成度报告 + 已交付功能清单 + 待规划功能

3. **可视化**：
   - 环节完成度仪表盘（每个环节一个进度环）
   - 解决方案导出（PDF/HTML）

**待沟通**：完成度评估的"预期功能点清单"从哪来？（商业地图源 Excel 是否已含规划项？）

## 六、关键风险与决策点

### 6.1 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 钉钉API权限申请失败或延迟 | A 方向卡壳 | 先支持Excel导入作为过渡方案 |
| 环节清单不存在或与商业地图不匹配 | B 方向卡壳 | B1 先定位，必要时安排张颖协作梳理 |
| 图片存储涉及 OSS 成本/权限 | A 方向卡壳 | 先用本地存储，规模上来再迁 OSS |
| 接入刘菡系统需重构 | A 方向返工 | A7 沟通时明确技术约束，预留重构空间 |
| 环节匹配准确率不足 | B3 数据质量差 | 人工种子集 + 规则 + 人工校验三道关 |
| HTML-to-Image 导出样式不一致 | 预览与最终图片差异大 | 使用相同HTML模板和CSS，Puppeteer设置正确视口 |

### 6.2 待决策点（待沟通后确认）

1. 后台技术选型最终形态（本地 Flask vs 完整 Web 应用）
2. 商业地图可视化形式（树状/流程/矩阵）
3. 环节清单的权威来源（项目映射表/季度规划/TAPD/张颖口述）
4. 完成度算法的"预期功能点"来源
5. 接入刘菡系统的技术方案
