## 1. 产品概述

### 1.1 背景与现状

发版路线图当前为数据驱动的静态 HTML（`window.RELEASE_DATA` 驱动），运营（王蓉）无法自主维护内容。现状问题集中在：

- **运营无法直接维护**：全流程依赖开发从钉钉文档拉取、跑脚本生成，运营是"需求方"而非"操作者"。
- **图片处理缺失**：源表"提供材料"列全空，整条管线不处理截图，王蓉需自行截图并人工拼长图。
- **环节维度为零**：发版记录无"环节"概念，无法支撑商业地图后续结合。
- **历史查询靠群聊翻找**：每次发版长图发到钉钉群，事后核对只能回翻群聊。

王蓉当前实际流程（已与王蓉、刘菡对齐）：

```
研发通知发版 → 整理到本地 Excel（含面向用户 / 是否重点等额外字段）
            → AI 按「面向用户」生成 3 张长图（总图 / 仅题单 / 仅教辅）
            → 分别发到不同钉钉群
```

### 1.2 核心痛点

| 痛点 | 现状 | 目标 |
|------|------|------|
| 数据整理 | 从钉钉文档手动复制到本地 Excel，再喂给 AI | 后台一键拉取并解析，自动填充 |
| 图片录入 | 自己截、自己拼，无系统承载 | 后台支持粘贴图片、按功能点关联 |
| 分版本生图 | 人工指挥 AI 出 3 张图，分别保存 | 后台按"面向用户"一键生成多版本长图并分别保存 |
| 权限控制 | 无，凭群聊天然隔离 | 公司要求做权限控制、按业务线数据隔离 |
| 历史查询 | 群聊翻找 | 后台按日期 / 类型 / 业务筛选查询 |
| 学习成本 | — | 王蓉明确"越简单越好" |

### 1.3 产品目标

1. 让运营**自主**维护发版内容、上传截图、生成并导出长图，不再依赖开发跑脚本。
2. 支持**按"面向用户"分版本生成长图**（总图 / 仅题单 / 仅教辅），分别保存与分发。
3. 建立**三层权限**：游客只读查看、管理员管配置、运营管内容，数据按业务线隔离。
4. 沉淀**历史发版数据**，支持筛选查询与统计，替代群聊翻找。

### 1.4 范围

**In Scope（本期）**
- 三层权限模型与本地账号体系。
- 管理员后台：数据源、排期、用户与权限、系统设置、审计、全局兜底。
- 运营后台：录入、列表、图片、分版本生图、预览导出、发布、统计。
- 游客公开查看端。
- 钉钉文档同步（先 Excel 导入过渡，再接 API）。

**Out of Scope（本期不做，已标注后续）**

- 对接公司 / 钉钉 SSO（MVP 用手动建账号）。
- 自动群分发（MVP 导出后运营自行发群，预留接口）。
- 商业地图（方向 B）的环节可视化与完成度评估。
- 公网部署（MVP 为本地 / 内网，部署升级后再公网公开游客端）。
- 环节视图（扫描 / 答题卡）与"功能怎么用"说明（闻道作业后续项）。

## 2. 用户角色与权限模型

### 2.1 三角色定义

| 角色 | 定位 | 能否编辑 | 数据范围 | 账号来源 |
|------|------|----------|----------|----------|
| **游客 Guest** | 只读查看端 | ❌ | 仅"已发布"内容（全产品线） | 无账号 |
| **管理员 Admin** | 系统配置端 | ⚙️ 配置类 + 全局兜底 | 全部（superuser） | 系统内置首个管理员，后续手动创建 |
| **运营 Ops** | 内容维护端 | ✅ | 仅自己被分配的产品线 | 管理员手动创建 |

### 2.2 RBAC 权限矩阵

| 能力域 | 游客 | 运营 | 管理员 |
|--------|------|------|--------|
| 查看已发布发版（路线图 / 详情） | ✅ | ✅ | ✅ |
| 录入 / 编辑 / 删除发版内容 | ❌ | ✅（仅自己产品线） | ✅（全部） |
| 触发钉钉同步 | ❌ | ✅（仅自己产品线） | ✅ |
| 配置钉钉源（URL/token/映射） | ❌ | ❌ | ✅ |
| 维护发版计划 / 排期 / 预告 | ❌ | ❌ | ✅ |
| 创建 / 禁用账号、分配角色 | ❌ | ❌ | ✅ |
| 管理产品线 / 运营-产品线绑定 | ❌ | ❌ | ✅ |
| 系统设置（模板 / 导出格式） | ❌ | ❌ | ✅ |
| 查看审计日志 | ❌ | ❌ | ✅ |
| 发布 / 撤回发版 | ❌ | ✅（仅自己产品线） | ✅ |
| 导出长图 | ❌ | ✅（仅自己产品线） | ✅ |
| 统计查询 | ✅（仅已发布） | ✅（仅自己产品线） | ✅（全部） |

### 2.3 数据隔离模型

- 隔离核心 = **运营 ↔ 产品线绑定**（管理员在后台配置）。
- 运营登录后，所有录入 / 列表 / 同步 / 统计均**自动限定**在其绑定产品线内；产品线字段在录入页呈锁定态（不可改）。
- 管理员可跨产品线查看 / 编辑 / 删除（兜底与配置需要），并承担运营误删等异常恢复。
- 游客仅能访问 `status = published` 的内容，天然不触及草稿与未发布数据。

## 3. 功能需求

### 3.1 游客端（公开查看端）

> 定位：公司内外可读的发版"结果页"。MVP 阶段升级现有双击 HTML（内网可见）；部署升级后转公网公开。

| 编号 | 功能 | 说明 |
|------|------|------|
| G-01 | 发版路线图总览 | Header 概览卡：本次发版 / 统计 / 产品线分布 / 下次发版 / 重要发版（P0需求） |
| G-02 | 发版列表浏览 | 日期 → 类型 → 条目，按产品线标签展示 |
| G-03 | 单条发版详情 | 含图片长图展示、功能点、入口、特殊说明 |

| G-06 | 访问限制 | 仅展示"已发布"内容；不提供源数据批量导出 |

### 3.2 管理员端（系统配置端）

| A-02 | 同步任务管理 | 手动触发、查看同步日志、失败重试 / 告警 |

#### B. 发版时间 / 排期管理
| 编号 | 功能 | 说明 |
|------|------|------|
| B-01 | 发版计划维护 | 各产品线预期发版时间、"下次发版"预告字段 |

#### C. 用户与权限管理（手动建账号）
| 编号 | 功能 | 说明 |
|------|------|------|

## 4. 关键业务流程

### 4.1 发版内容维护与发布（主流程）

```
运营登录 → 钉钉文档一键拉取（O-01，限自己产品线）
        → 补充字段（O-02：面向用户 / 是否重点 / 类型 / 涉及业务 / 通知时间…）
        → 粘贴 / 上传截图（O-05/O-06）
        → 保存草稿（O-04）
        → 预览（O-11）→ 分版本生图（O-07/O-08）→ 逐张预览（O-10）
        → 导出长图（O-12）
        → 设定发布时间（O-14）→ 直接发布（O-15，无审批）
        → 游客端可见（G-03）
```

 管理员登录  → 钉钉文档一键拉取（O-01，可以对所有产品线操作）
         → 补充字段（O-02：面向用户 / 是否重点 / 类型 / 涉及业务…）
         → 粘贴 / 上传截图（O-05/O-06）
         → 保存草稿（O-04）
         → 预览（O-11）→ 分版本生图（O-07/O-08）→ 逐张预览（O-10）
         → 导出长图（O-12）
         → 设定发布时间（O-14）→ 直接发布（O-15，无审批）
         → 游客端可见（G-03）
 ~~~~



### 4.2 钉钉文档同步流程

- **配置侧（管理员）**：A-01 配置源 URL / token / 字段映射，绑定产品线。
- **触发侧（运营）**：O-01 触发同步，系统解析表格（系统 / 功能入口 / 功能点）自动填充到录入草稿。
- **过渡方案**：钉钉 API 权限未到位时，A-03 Excel / OCR识别截图导入等效替代；运营仍走 O-01 入口导入本地 Excel。

### 4.3 按"面向用户"多版本长图生成（核心）

```
同一批发版功能点（含 target_user 字段）
   ├─ 全量 → 总图（题单+教辅）
   ├─ 过滤 target_user ∈ {题单, 全部} → 仅题单用户图
   └─ 过滤 target_user ∈ {教辅, 全部} → 仅教辅用户图
每张图独立渲染、独立保存（O-09）、独立导出（O-12）
```

> 该能力对应王蓉现流程"AI 按 3 类用户出 3 张图分别发群"的刚需，是运营后台与原 v1 规划的最大差异点。

### 4.4 累积 / 合并发布流程

- 小优化频发，单次不值得单独发图 → 存草稿（O-04）。
- 草稿可累积，到下个发版批次统一发布（B-01 计划时间驱动）。
- 在编辑时自动填充下个批次的发版时间，并生成最新长图，避免反复重做。

### 4.5 数据隔离与权限校验

```
请求到达 → 鉴权（游客 / 运营 / 管理员）
  ├─ 游客：仅返回 status=published
  ├─ 运营：SQL 自动追加 WHERE product_line_id = 当前运营绑定项
  └─ 管理员：无产品线限制；E-01 可跨线兜底
```

## 5. 数据模型（关键实体）

> MVP 用 SQLite；以下为关键表与字段（字段命名见《发版记录版本优化规划》数据结构扩展）。

| 表 | 关键字段 | 说明 |
|----|----------|------|
| `users` | id, username, password_hash, role[admin/ops], product_line_id, status | 运营账号由管理员创建；游客无记录 |
| `product_lines` | id, name | 闻道作业 / 直播教学（可扩展） |
| `releases` | id, release_id, date, pl, type, fp, fe, module, scope, special, limit, is_highlight, target_user, verify_conclusion, images[JSON], operator, status[draft/published], environment, created_at, updated_at | 单条发版记录；target_user 为分图核心字段 |
| `data_sources` | id, name, type[dingtalk/excel/screenshot], url, token, product_line_id, field_mapping[JSON], sync_status, last_sync_at | 钉钉源配置（管理员） |
| `sync_logs` | id, source_id, status, message, at | 同步日志 |
| `audit_logs` | id, user_id, action, target, at | 审计留痕 |

**字段说明补充**
- `target_user`（面向用户）：题单 / 教辅 —— 分图过滤主键。
- `is_highlight`：是否重点（P0）。
- `environment`（环节）：后续项（扫描 / 答题卡），MVP 预留字段。
- `images`：JSON 数组，存图片 URL + 关联功能点 + 排序。

## 6. 非功能性需求

| 类别 | 要求 |
|------|------|
| **安全** | 密码哈希存储；session 鉴权；运营越权访问他人产品线须在 SQL 层强制拦截（非仅 UI 隐藏） |
| **审计** | 关键操作（登录 / 创建 / 编辑 / 同步 / 发布 / 删除）留痕（D-03） |
| **易用性** | 运营录入页极简；分图生成默认一键出全版本，通过按钮/键盘左、右切换不同版本的图片（王蓉"越简单越好"） |
| **兼容性** | 长图样式与现有长图风格一致；导出 PNG |
| **性能** | 本地单用户场景，长图渲染 < 数秒；数据量千级无压力 |
| **可扩展性** | 产品线 / 角色预留扩展；图片存储预留 OSS；群分发预留接口 |

---

## 7. 技术架构方案

### 7.1 MVP：本地 Web 原型

- **后端**：Flask + SQLite。
- **认证**：session + 密码哈希；管理员手动建账号（不接 SSO）。
- **前端**：游客页 = 现有 `发版路线图.html` 升级为模板化只读渲染；后台 = 表单 / 列表页（沿用产品看板 V1.4 视觉：深色侧边栏 + 浅色主区 / 蓝紫主色）。
- **钉钉同步**：钉钉文档 API（需申请权限）；MVP 先 Excel 导入过渡（A-03）。
- **导出**：Playwright / Puppeteer 将预览页截为长图。
- **图片**：MVP 存本地 `uploads/`，生成可访问 URL。

### 7.2 部署升级路线（未来）

- 功能验证完 → 评估升级为可部署 Web 应用。
- 游客端转**公网公开**（最终态）；后台仍限内网 / 登录。
- 预留：公司 / 钉钉 SSO、OSS 存储、群分发、与刘菡系统对接（方向 A7）。
