### 1.2 问题诊断

#### 问题 1：P4（管理员账号管理）滞后导致依赖断裂

**严重程度**：🔴 高

P0 只建立了三角色登录/会话骨架，但除非系统内置了首个管理员账号 + 至少一个运营测试账号，否则 P1 的运营端闭环根本无法被任何人登录使用。

**建议**：
- **方案 A（推荐）**：将 P4 中「手动建账号 + 角色分配」的精简版提前到 P0 末尾或 P1 开头，作为 P0.5：「管理员初始化」—— 0.3 周完成：内置 admin 账号 + 管理员可创建首个运营账号。
- **方案 B**：P0 中内置硬编码的 admin/wangrong 测试账号，P4 再做完整账号管理。

#### 问题 2：P3 与 P1 存在逻辑耦合

**严重程度**：🟡 中

P3 的「通知时间设置」「发版日期只读」是 P1 编辑表单的一部分（PRD §3.3 O-04 编辑必填字段表中已列出）。将表单字段拆到两个阶段会导致 P1 交付一个"字段不全"的编辑页，P3 再做补丁式添加，增加返工。

**建议**：将 P3 中「通知时间 + 发版日期只读」合并到 P1（编辑表单本身就有这些字段）。P3 仅保留「发布后更正/补充」独立交付（0.3 周）。

#### 问题 3：P5 配置项可与 P4/P6 并行

**严重程度**：🟢 低

P5（发布清单位置配置、导入通道、系统设置、审计日志）与 P4（账号管理）无硬依赖，且管理员配置类功能可独立验证。可考虑 P4 与 P5 部分并行。

## 2. 关键里程碑定义

### 里程碑 M1：运营可自主完成一次发版（第 3 周结束）

**定义**：王蓉登录系统 → 从钉钉文档发布清单导入 → 编辑发版详情 → 粘贴截图 → 保存草稿 → 发布 → 游客端可见。

**通过标准**：
- [ ] 王蓉使用自己的账号登录运营后台（即使是硬编码账号）。
- [ ] 从钉钉文档发布清单导入 ≥5 条功能点，数据正确填充到编辑表单。
- [ ] 编辑至少 3 条功能点，补充「面向用户」「是否重点」「发版类型」字段。
- [ ] 在 2 条功能点下 Ctrl+V 粘贴截图（≥2 张），每张响应 <200ms。
- [ ] 保存草稿后退出，重新登录草稿仍在。
- [ ] 设置通知时间并发布，游客端可立即看到新增内容。
- [ ] 完成一次完整发布 **≤5 步**（导入→编辑→生图→预览→发布）。

**汇报要点**：「运营闭环已打通——从钉钉文档发布清单导入到游客端可见，全流程可由运营自主完成，无需开发介入。」

---

### 里程碑 M2：分版本长图生成可用（第 5 周结束）

**定义**：王蓉一键生成 3 张分版本长图（总图/作业/教辅），可预览切换、逐张下载。

**通过标准**：
- [ ] 基于同一批发版数据，一键生成「总图」「仅作业」「仅教辅」3 张 PNG。
- [ ] 3 张图风格 100% 一致（模板锁定 + 数据驱动渲染）。
- [ ] 单张渲染 <5 秒，3 张并行 <10 秒。
- [ ] 预览区可通过 Tab/左右箭头切换版本。
- [ ] 可逐张下载为 PNG，也可全部打包下载（ZIP）。
- [ ] 当某版本无功能点时（如全部标记为作业），该版本图提示「无相关功能点」。
- [ ] 生图结果与现有长图风格对照：王蓉确认风格可接受。

**汇报要点**：「核心差异能力已就绪——一键生成面向不同受众的 3 张版本长图，风格统一、操作极简。」

**定义**：管理员可管理账号/配置/全局兜底，王蓉试用全流程并通过体验验收。

**通过标准**：
- [ ] 管理员可创建/禁用运营账号，分配产品线，数据隔离生效（运营 A 看不到运营 B 的数据）。
- [ ] 管理员可配置钉钉文档发布清单地址与字段映射，运营可成功导入。

- [ ] 发布后更正/补充：对已发布内容追加更正信息，重新生图，更正标记清晰可见。
- [ ] 王蓉明确表示「可以用」或等效确认。

**汇报要点**：「发版路线图系统 V2 已就绪，运营可独立完成从导入到分版生图的全流程，管理员具备完整管控能力。」

### 3.3 P2 分版本生图：PoC 先行建议

#### 为什么需要 PoC

P2 是整个系统技术风险最高的模块，原因：

1. **渲染方案选型不确定**：Playwright 和 Puppeteer 在长图截取场景下各有优劣。Playwright 对中文渲染更好但安装体积大，Puppeteer 轻量但中文 font-family 可能有问题。
2. **模板 + 数据驱动渲染的可行性**：HTML 模板填充动态数据后用无头浏览器截长图，需要验证：
   - 中文排版（字体回退、行高、标点）一致性
   - 图片嵌入（运营粘贴的截图在长图中的呈现）
   - 长图尺寸（无限高截取方案的性能）
3. **性能目标达成风险**：单张 <5 秒、3 张并行 <10 秒，在 Python/Flask 进程中调用无头浏览器的开销未知。
4. **风格一致性验证**：模板锁定的风格是否能 100% 与现有长图风格一致。

#### PoC 方案

| 项目 | 内容 |
|------|------|
| **目标** | 验证「HTML 模板 + 数据填充 + Playwright 截长图」技术路线的可行性 |
| **周期** | **0.3 周（约 1.5 天）** |
| **输入** | 一份模拟发布清单数据（≥5 条功能点）+ 2-3 张模拟截图 |
| **产出** | 1 张模拟长图 PNG（含功能点 + 截图嵌入），风格与现有长图对照 |
| **通过标准** | ① 渲染正确（中文、图片、布局） ② 渲染时间可接受 ③ 风格与现有长图基本一致 |

**PoC 建议位置**：P1 结束前启动（此时已有真实数据结构和字段），与 P1 末尾并行，不阻塞 P1 交付。

#### PoC 失败降级方案

若 Playwright 路线不可行：
- **降级 A**：改用 Pillow 纯 Python 图片合成（放弃 HTML 模板，用代码拼接图片）。优点：无外部依赖、性能可控。缺点：模板调整灵活性差。
- **降级 B**：预渲染 HTML → 用户手动在浏览器中「打印为 PDF/截图」。优点：最简。缺点：步骤数增加，违背 ≤5 步目标。
