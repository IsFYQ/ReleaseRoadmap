## 1. 产品概述

### 1.1 背景与现状

#### 现状流程（完整端到端）

发版路线图当前为数据驱动的静态 HTML（`window.RELEASE_DATA` 驱动），运营（王蓉）无法自主维护内容。完整业务流程如下：

```
PM 通知发版 → 产品团队出发布清单（含功能点、功能入口、涉及端、特殊说明等）
            → 运营验证功能点（在发布清单基础上扩展验证结论、截图等）
            → ★ 系统入口 ★ → 运营导入已验证功能点
            → 编辑发版详情（补充面向用户、是否重点、粘贴截图）
            → 长图预览 → 分版本生图 → 下载 / 发布
```

**V1.0 的核心偏差**：将"钉钉文档拉取"作为流程第一步，且未区分"发布清单"与原始文档。实际情况是：
- **发布清单**：由产品团队产出并维护在**钉钉文档**中，含功能点、功能入口、涉及端、特殊说明等字段，是运营验证的**输入**。
- **验证清单**：运营在发布清单基础上扩展（增加截图、面向用户、通知时间等），是系统的**入参**。
- 系统应同步的数据源是**钉钉文档中的发布清单**，运营端导入时只需拉取发布清单数据并扩展通知时间、面向用户等字段。

#### 王蓉当前实际流程

```
研发通知发版 → 从钉钉文档手动复制到本地 Excel（含面向用户 / 是否重点等额外字段）
            → 自行截图、添加红色箭头标注
            → 喂给 AI 按「面向用户」生成 3 张长图（总图 / 仅作业 / 仅教辅）
            → 分别发到不同钉钉群
```

#### 痛点量化

| 痛点 | 现状 | 量化估计 | 目标 |
|------|------|----------|------|
| 数据整理 | 从钉钉文档手动复制到本地 Excel，再喂给 AI | 每次发版约 8-10 步操作 | 系统导入，≤5 步完成 |
| 图片录入 | 自己截、自己拼，无系统承载 | 每功能点手动截图→标注→粘贴 | 粘贴即上传，<200ms 响应 |
| 分版本生图 | 人工指挥 AI 出 3 张图，分别保存 | 3 次独立 AI 对话 + 3 次保存 | 一键生成，自动拆分 |
| 权限控制 | 无，凭群聊天然隔离 | — | 按业务线数据隔离（公司要求） |
| 历史查询 | 群聊翻找 | 平均 >2 分钟定位一条 | 按日期/类型/业务筛选，<10 秒 |
| 学习成本 | — | 王蓉明确"越简单越好" | 零术语、零配置 |

### 1.2 核心痛点

同上表，核心矛盾在于：**王蓉需要的是"信息进来 → 图片出去"的最短路径**，而现状是 8-10 步的手动流水线。

### 1.3 产品目标

1. 让运营**自主完成"导入 → 编辑 → 生图 → 下载"全流程**，不再依赖开发跑脚本。目标：≤5 步完成一次发版。
2. 支持**按「面向用户」一键分版本生成长图**（总图 / 仅作业 / 仅教辅），风格锁死、所见即所得。
3. 运营**自主控制通知时间与是否生成图片**，发版时间由研发侧确定、运营端只读展示。
4. 建立**三层权限**：游客只读查看、管理员管配置、运营管内容，数据按业务线隔离。
5. 沉淀**历史发版数据**，支持筛选查询与统计，替代群聊翻找。
6. 支持**发布后"更正/补充"**能力：已发布内容可追加修正信息，而非覆盖原内容。

### 1.4 范围

**In Scope（本期 V2）**

- 三层权限模型与本地账号体系。
- 管理员后台：数据源、排期、用户与权限、系统设置、审计、全局兜底。
- **运营后台（重点修正）**：
  - **发布清单导入**（从钉钉文档中的发布清单拉取，并扩展通知时间、面向用户等字段；支持手动录入作为补充通道）——替代 V1.0 的"本地 Excel 手动导入"。
  - 发版详情编辑（字段去验证化，统一为发版详情编辑语境）。
  - Ctrl+V 粘贴图片（P0 体验要求，<200ms 响应）。
  - 一键分版本生图 + 版本切换预览 + 逐张导出。
  - 发版时间只读展示，通知时间运营可控。
  - 发布后"更正/补充"能力。
- 游客公开查看端。
- 发布清单同步（从钉钉文档中的发布清单拉取）。

**Out of Scope（本期不做，已标注后续）**

- 对接公司 / 钉钉 SSO（MVP 用手动建账号）。
- 自动群分发（MVP 导出后运营自行发群，预留接口）。
- 商业地图（方向 B）的环节可视化与完成度评估。
- 公网部署（MVP 为本地 / 内网，部署升级后再公网公开游客端）。
- 环节视图（扫描 / 答题卡）与"功能怎么用"说明（闻道作业后续项）。
- 审批流（MVP 直接发布无审批，预留 LaunchNotes 式草稿→审核→发布架构）。
- Widget 嵌入模式（预留 Canny 式一套数据多视图呈现架构）。

### 1.5 产品定位

基于竞品分析结论，本产品定位为**发版可视化公告引擎**——市场空白品类。完整链路为：

> 发版管理 → 按受众分版本 → 自动生成长图 → 公开页发布

目前没有任何竞品实现此完整链路。三大差异化：
1. **端到端闭环**：发版管理 + 长图生成一站式。
2. **受众维度创新**：教育内容场景特有的「作业 / 教辅 / 全部」分版逻辑。
3. **双模架构**：内部运营工具 + 外部公开页。

### 2.4 权限边界（关键修正）

**发版时间权限**（V2 明确）：
- 发版时间**完全由研发控制**，通过发布清单带入系统。
- 运营端**仅展示不可编辑**。
- 运营唯一可控的时间字段是「**通知时间**」（即何时向游客端公开）。

**生图权限**：
- 运营自主决定「是否生成图片」以及「何时生成」。
- 管理员可配置长图模板与风格参数（主题色等）。

### 2.5 数据隔离模型

- 隔离核心 = **运营 ↔ 产品线绑定**（管理员在后台配置）。
- 运营登录后，所有导入 / 编辑 / 列表 / 生图均**自动限定**在其绑定产品线内；产品线字段在录入页呈锁定态（不可改）。
- 管理员可跨产品线查看 / 编辑 / 删除（兜底与配置需要），并承担运营误删等异常恢复。
- 游客仅能访问 `status = published` 的内容，天然不触及草稿与未发布数据。

### 3.3 运营端（内容维护端，按产品线隔离）★ 重点修正

- **运营管"内容"，越权不到他人产品线。**

- **零术语**：所有字段使用运营日常用语，无"验证"等技术色彩。
- **粘贴即上传**：Ctrl+V 直接粘贴截图，<200ms 响应。

#### 一、发布清单导入（★ V2 修正：从钉钉文档发布清单拉取，扩展通知时间、面向用户）

| 编号 | 功能 | 说明 |
|------|------|------|
| O-01 | **发布清单导入** | 从钉钉文档中的发布清单拉取已验证功能点，自动填充：发版日期、功能点描述、功能入口、涉及应用/平台、涉及端、特殊说明、限制条件、是否重点、备注；运营补充**通知时间**与**面向用户**后进入编辑。导入地址由管理员在 A-01 配置。 |
| O-02 | 手动录入功能点 | 不强制依赖导入，可独立逐条录入功能点（作为导入的补充通道） |
| O-03 | 功能点手工增删改 | 导入后可增删改单条功能点，支持调整排序 |

**验收标准（O-01）**：

- [ ] Given 管理员已配置钉钉文档发布清单地址，When 运营点击「导入发布清单」，Then 系统从钉钉文档拉取发布清单并自动填充功能点列表，包含：功能点描述、功能入口、涉及应用/平台、涉及端、特殊说明。
- [ ] Given 导入成功，When 查看功能点列表，Then 每条功能点显示导入来源（钉钉文档发布清单 + 导入时间）。
- [ ] Given 钉钉文档发布清单格式不符合模板，When 点击导入，Then 系统提示具体格式错误（如"缺少'功能点描述'列"），不静默失败。
- [ ] Given 同一发布清单重复导入，When 系统检测到内容重复，Then 提示"该发布清单已导入过，是否覆盖？"。

**发布清单字段映射表（已确认）**：

> 发布清单为钉钉文档中的结构化表格，包含以下字段。系统导入时按下方映射自动填充到发版详情草稿。

| 发布清单字段 | 系统字段 | 说明 | 运营可编辑 |
|--------------|----------|------|------------|
| 发版时间 | 发版日期 | 只读展示，运营不可修改 | ❌ |
| 平台/范围 | 平台/范围 | 描述该功能所属平台或影响范围 | ✅ |
| 应用或后台 | 涉及应用/平台 | 学生端App / 教师端后台 / 管理后台等 | ✅ |
| 端 | 涉及端 | PC / App / 小程序 | ✅ |
| 新增、优化、修复 | 发版类型 | 新增 / 优化 / 修复 | ✅ |
| 需验证功能点 | 功能点描述 | 对该功能点的简要描述 | ✅ |

## 4. 关键业务流程

### 4.1 发版内容维护与发布（主流程 — ★ V2 修正）

```
PM通知发版
  → 产品团队输出发布清单（钉钉文档，含功能点描述/功能入口/涉及应用/平台/涉及端/特殊说明）
      → 运营基于发布清单验证功能点、截图
          → ★ 系统入口 ★
              → 运营导入已验证功能点（O-01：从钉钉文档发布清单拉取，或手动录入）
                  → 编辑发版详情（O-04：补充面向用户/通知时间/是否重点/发版类型，发版日期只读）
                      → Ctrl+V 粘贴截图（O-05/O-06）
                          → 保存草稿（O-04a）
                              → 一键分版本生图（O-07：总图/仅作业/仅教辅）
                                  → 版本切换预览（O-08）
                                      → 逐张导出/全部下载（O-09/O-12）
                                          → 设置通知时间（O-13）
                                              → 直接发布（O-14）
                                                  → 游客端可见（G-03）
                                                      → 发布后更正/补充（O-15）
```

**步骤数分析**：

| 步骤 | 操作 | 累计 |
|------|------|------|
| 1 | 导入发布清单（一键） | 1 |
| 2 | 编辑发版详情（批量编辑 + 粘贴截图） | 2 |
| 3 | 一键分版本生图 | 3 |
| 4 | 预览确认 + 下载 | 4 |
| 5 | 设置通知时间 + 发布 | 5 |

**目标达成**：≤5 步（vs 现状 8-10 步，减少 ≥37.5%）。

### 4.2 发布清单同步流程（★ V2 修正：从钉钉文档发布清单拉取）

- **配置侧（管理员）**：A-01 配置钉钉文档发布清单 URL / 地址、字段映射，绑定产品线。字段映射以「发布清单字段映射表」为默认模板，管理员可按实际表格微调。
- **导入侧（运营）**：O-01 触发导入，系统从钉钉文档拉取发布清单（发版日期 / 功能点描述 / 功能入口 / 涉及应用/平台 / 涉及端 / 特殊说明 / 限制条件 / 是否重点 / 备注）自动填充到发版详情草稿；运营补充通知时间、面向用户等字段。
- **过渡方案**：发布清单格式标准化之前，O-02 支持手动逐条录入。

### 4.3 按"面向用户"多版本长图生成（核心）

```
同一批发版功能点（含 target_user 字段）
   ├─ 全量 → 总图（作业 + 教辅 + 全部）
   ├─ 过滤 target_user ∈ {作业, 全部} → 仅作业用户图
   └─ 过滤 target_user ∈ {教辅, 全部} → 仅教辅用户图
每张图独立渲染、独立保存（O-09）、独立导出（O-12）
```

> 该能力对应王蓉现流程"AI 按 3 类用户出 3 张图分别发群"的刚需。得益于模板+数据驱动的渲染架构（参考 Pixelixe 模式），模板锁定风格，数据动态填充，确保 3 张图风格一致。

### 4.4 累积 / 合并发布流程

- 小优化频发，单次不值得单独发图 → 存草稿（O-04a）。
- 草稿可累积，到下个发版批次统一发布（B-01 计划时间驱动）。
- 在编辑时自动填充下个批次的发版时间，并生成最新长图，避免反复重做。

### 4.5 发布后更正/补充流程（★ V2 新增）

```
已发布发版（status=published）
  → 运营点击「更正/补充」
      → 系统进入追加编辑模式（原始内容锁定，新增更正/补充区域）
          → 运营填写更正/补充内容
              → 保存（状态不变，增加 correction 标记）
                  → 重新生成长图（自动在顶部追加更正/补充区块）
                      → 预览确认 → 替换发布内容
```

### 4.6 数据隔离与权限校验

```
请求到达 → 鉴权（游客 / 运营 / 管理员）
  ├─ 游客：仅返回 status=published
  ├─ 运营：SQL 自动追加 WHERE product_line_id = 当前运营绑定项
  └─ 管理员：无产品线限制；E-01 可跨线兜底
```

## 6. 非功能性需求（★ V2 强化）

| 类别 | V2 要求 | V1 对照 |

### 体验风险防控（来自用户研究）

| 致命风险 | 防控措施 | 验收手段 |
|----------|----------|----------|
| 步骤数无显著减少 | 设计目标 ≤5 步，每步操作极简（导入一键、生图一键） | 可用性测试：王蓉首次使用计时 |
| 图片风格偏差 | 模板锁定 + 数据驱动渲染，风格由模板决定而非数据 | 对比测试：生成图 vs 现有长图风格 |
| 粘贴性能差 | <200ms 响应，前端本地预览 + 异步上传 | 性能测试：连续粘贴 5 张大图 |

## 7. 技术架构方案

### 7.1 MVP：本地 Web 原型

- **后端**：Flask + SQLite。
- **认证**：session + 密码哈希；管理员手动建账号（不接 SSO）。
- **前端**：游客页 = 现有 `发版路线图.html` 升级为模板化只读渲染；后台 = 表单/列表页（沿用产品看板 V1.4 视觉：深色侧边栏 + 浅色主区 / 蓝紫主色）。
- **发布清单导入**：钉钉文档 API 拉取并解析（或 openpyxl / pandas 解析导出的离线文件作为过渡），替代 V1 的本地 Excel 手动导入。
- **长图生成**：模板 + API 驱动渲染架构（参考 Pixelixe 模式）。模板 HTML 定义品牌风格，数据通过 API 动态填充，Playwright / Puppeteer 截为长图。
- **图片粘贴**：前端监听 paste 事件 → FileReader 本地预览 → FormData 异步上传，确保 <200ms 响应。
- **图片存储**：MVP 存本地 `uploads/`，生成可访问 URL。

### 7.2 部署升级路线（未来）

- 功能验证完 → 评估升级为可部署 Web 应用。
- 游客端转**公网公开**（最终态）；后台仍限内网 / 登录。
- 预留：公司 / 钉钉 SSO、OSS 存储、群分发、审批流、Widget 嵌入模式（Canny 式）。
